## Problem

High-demand restaurants release reservations at specific times, and the window to book one is measured in seconds. I wanted to build a system that could monitor availability across multiple users, detect openings the moment they appeared, and execute bookings faster than a person could click through a website. The challenge was not just speed — it was reliability at scale. When fifty users are monitoring simultaneously, the system needs distributed coordination to prevent double-bookings, handle upstream API changes gracefully, and avoid detection by the reservation platform.

## Solution & Approach

I architected SeatWatch as a Turborepo monorepo with three independently deployed services: a React SPA for the user dashboard, an Express REST API handling 40+ endpoints, and a BullMQ worker service processing five concurrent job queues. The separation means each service scales independently — the worker can be horizontally scaled without touching the API or frontend.

The core of the system is a dual-strategy booking engine. The primary path makes direct API calls to the restaurant platform, executing in under a second. When the API returns unexpected errors indicating compatibility issues, the system automatically falls back to a headless browser session using Patchright, a Playwright fork optimized for automation. The browser path is slower but resilient to API changes since it operates through the same interface a human would use.

For concurrent safety, I implemented distributed booking locks using Redis `SET NX EX` with UUID nonces and Lua-script-safe release. This prevents two worker instances from double-booking when they detect the same availability simultaneously. A per-user circuit breaker tracks failures and automatically pauses monitoring when a user's account hits rate limits, with exponential cooldown and probe-based recovery.

Detection evasion required more engineering depth than I anticipated. Each user is deterministically assigned a persistent browser identity — User-Agent, TLS fingerprint, viewport dimensions, cookie jar — that stays consistent across all their sessions. Poll intervals use Poisson-distributed stagger offsets so that multiple users monitoring the same venue do not create correlated request patterns.

## Tech Stack Detail

I chose TypeScript across the entire stack for type safety from the React frontend through the Express API to the worker processes. BullMQ handles job queuing with five dedicated queues, each with independent concurrency tuning — the monitor queue runs at concurrency 10 for availability polling, while the book queue runs at 5 to limit concurrent reservation attempts.

PostgreSQL stores all application data with Prisma as the ORM, and Redis serves double duty as the BullMQ backend and the shared state store for circuit breakers, rate limiters, and booking locks. Stripe powers a four-tier billing system with transaction-level plan enforcement — the API checks active request counts within a database transaction to prevent race conditions from concurrent request creation.

User credentials for the restaurant platform are encrypted at rest using AES-256-GCM with per-record initialization vectors. Credentials are decrypted only in worker memory at the moment of use and never appear in API responses or logs.

## Challenges & Lessons

The hardest problem was the distributed booking lock. My first implementation used a simple Redis `SET NX` without a nonce. This worked until I hit the scenario where a slow worker's lock expired, another instance acquired it, and the original worker deleted the new lock on completion. The fix was atomic nonce-checked deletion via a Lua script — a textbook distributed systems problem that I had to encounter firsthand to fully understand.

The circuit breaker design went through three iterations. The first was in-memory and lost state on worker restart. The second used Redis but did not handle the half-open recovery probe correctly. The final implementation is a proper state machine (CLOSED, OPEN, HALF_OPEN) stored in Redis hashes, surviving restarts and working correctly across multiple worker instances.

I also learned that detection evasion is fundamentally about consistency, not randomness. Randomizing everything makes traffic look more suspicious, not less. A real user has a consistent browser fingerprint, consistent request patterns, and consistent session behavior. The system works because each user's automation looks like one person using a browser normally.

## Results

SeatWatch handles approximately fifty parallel user sessions across three deployed services on Railway. The monitoring system processes thousands of availability checks daily with jittered scheduling that keeps request patterns indistinguishable from organic traffic. The sniping mode lands first-poll attempts within milliseconds of reservation release times. The codebase spans 48,000 lines of TypeScript across 329 files with 90+ test files. The billing system has processed real transactions through all four plan tiers without a race condition or double-charge.
