## Problem

When I started applying for junior software engineering roles, I noticed a pattern. My resume listed technologies and bullet points, but it never told the full story. Recruiters would glance at it for thirty seconds, and there was no way to convey the depth of thought behind any of my projects in that format. I needed something that could work at two speeds: fast enough for a recruiter scanning dozens of candidates, and deep enough for an engineer who wanted to understand how I think about building software.

A portfolio site seemed like the obvious answer, but I wanted it to be more than a list of links. I wanted the site itself to demonstrate the same skills I was trying to showcase. If I was claiming to care about performance, the site had better be fast. If I was claiming to write clean code, the codebase had better reflect that. The portfolio had to be its own best case study.

## Solution & Approach

I approached this the way I would approach any production project. I started with research into modern static site frameworks, evaluating trade-offs between developer experience, performance, and long-term maintainability. I chose Astro 6 because its zero-JavaScript-by-default architecture meant every page ships as pure HTML unless I explicitly opt into client-side interactivity. For a content-focused portfolio, that philosophy aligned perfectly with what I needed.

The site architecture follows a content-driven approach. Project case studies live as MDX files with typed frontmatter schemas validated by Zod at build time. This means a broken project entry never makes it to production. Each project has structured metadata like tech stack, category, featured status, and links to source code or live demos. The content layer handles all of this with compile-time type safety.

For styling, I built a design token system on top of Tailwind CSS v4. Every color, spacing value, and typography scale is defined as a CSS custom property that flows through the entire site. This means the visual system is consistent and maintainable. Changing the accent color or adjusting spacing proportions is a single-token edit, not a search-and-replace across dozens of files.

The layout system follows an editorial pattern inspired by high-end design portfolios. Section headers use monospaced uppercase labels on the left with content flowing in the right column of an asymmetric grid. This creates a clear visual hierarchy without relying on heavy borders or background colors.

## Tech Stack Detail

I selected each technology based on specific requirements rather than familiarity. Astro 6 was chosen for its static-first architecture and built-in content collection system. The Content Layer API with glob loader means adding a new project is as simple as dropping an MDX file into a directory. TypeScript provides type safety across the entire codebase, from component props to content schemas. The Zod schema validation catches content errors at build time rather than surfacing them as runtime bugs.

Tailwind CSS v4 was the styling choice because of its CSS-first configuration approach. The v4 Rust rewrite with the Oxide engine made builds noticeably faster, and the ability to define the entire design system in CSS using the @theme directive eliminated the need for a separate JavaScript configuration file. Every utility class references design tokens, keeping the styling layer tightly integrated with the visual system.

MDX powers the case study pages, allowing me to mix prose content with custom Astro components when needed. Images are processed through Astro's built-in image optimization pipeline, which automatically generates WebP and AVIF formats with proper srcset attributes for responsive loading.

Deployment targets Cloudflare Pages for its unlimited bandwidth on the free tier, global edge network, and first-class Astro support. The build-deploy pipeline is simple: push to main, Cloudflare builds and deploys within minutes.

## Challenges & Lessons

The biggest challenge was resisting the urge to over-engineer. Early on, I considered adding client-side filtering for the projects page, animated page transitions with GSAP, and a theme switcher. I had to step back and ask what actually served the goal. The goal was to get interviews, not to build the most technically impressive portfolio possible. Features that did not directly help a recruiter or engineer evaluate me as a candidate got deferred.

Another challenge was writing about my own work. It is surprisingly difficult to explain technical decisions in a way that is honest without being either self-deprecating or self-promotional. I settled on a first-person conversational tone that focuses on the why behind each decision. I talk about what I considered, what I chose, and what I learned. That felt more authentic than listing features.

I also learned the value of content schemas. When I first set up the project collection, I made the thumbnail field required. This meant every placeholder project needed a fake image. Making the field optional and using a solid-color fallback was a small schema change that eliminated an entire category of maintenance burden. Small decisions like that compound.

## Results

The site achieves perfect or near-perfect Lighthouse scores across all categories. Every page loads in under one second on a broadband connection because there is zero client-side JavaScript on static pages. The content system supports adding new projects in minutes. The design token system makes visual updates predictable and consistent. Most importantly, the site tells a story that a resume cannot: here is how I think, here is how I build, and here is what I care about.
