## Problem

I wanted to build a prediction system that went beyond what you see in most ML tutorials. The standard approach — scrape some data, train a model on a random train/test split, report an accuracy number — teaches you scikit-learn's API but teaches you nothing about building a system you can actually trust. The fundamental challenge in sports prediction is temporal: you cannot use Thursday's closing line to predict Wednesday's game, and random cross-validation will silently let you do exactly that. I wanted a pipeline where future data leakage was structurally impossible, not just something I tried to avoid.

## Solution & Approach

The system is built around a three-layer data pipeline (Bronze/Silver/Gold) with Pydantic validation at every boundary. Bronze stores append-only raw snapshots from three external sources — game data via nflreadpy, odds via The Odds API, and weather via Open-Meteo. Silver stores cleaned, deduplicated tables. Gold stores model-ready feature matrices with 55-70 engineered features per game.

Temporal safety is enforced at three levels. Every feature builder implements a protocol with an `as_of_datetime` parameter — data after the cutoff is structurally inaccessible. A LeakageGate validator scans the assembled feature matrix for leakage keywords like `closing` or `final_score` and hard-fails if any slip through. All evaluation uses expanding-window walk-forward splits across 2021-2024 holdout seasons, never random cross-validation.

I train three specialized models — logistic regression with isotonic calibration for win probability, and XGBoost regressors for spread and total predictions. Rather than treating model outputs and market prices as competing signals, I blend them in mathematically appropriate spaces: log-odds for probabilities, linear interpolation for point values. Blend weights are tuned on temporally isolated pre-backtest data so they never see the evaluation window.

The web dashboard is a FastAPI application serving HTML pages through Jinja2 with HTMX for interactivity. Week switching, sort controls, and season filtering update the page without full reloads. Game detail pages show prediction-versus-market comparisons, feature importance charts, and team context. The web layer reads exclusively from a denormalized DuckDB cache — no model imports, no training code. The API starts in milliseconds and cannot be broken by changes to the ML pipeline.

## Tech Stack Detail

Python was the natural choice given the ML ecosystem. DuckDB paired with Parquet gives me SQL query flexibility for ad-hoc analysis and columnar read performance for the training pipeline. Feature engineering uses pandas with custom rolling-window calculations for Elo ratings, EPA metrics from play-by-play data, and opponent-adjusted statistics.

The Elo system implements margin-of-victory K-factor adjustment with Glicko-style uncertainty tracking and a divisional home-field advantage reduction factor based on research showing divisional games exhibit roughly 46% less HFA. Market features include devigged moneyline probabilities and line movement tracking. Weather features compress 38 raw columns into 4-6 dominant features through categorical bucketing and impact scoring.

FastAPI with HTMX was a deliberate choice over a JavaScript SPA. The data is read-only and changes once per week. Server-rendered HTML with targeted fragment swaps is faster to build, faster to load, and requires zero client-side state management. Plotly handles the interactive charts for calibration plots, cumulative CLV curves, and betting equity simulations.

## Challenges & Lessons

The biggest lesson was how easy it is to leak future data without realizing it. My first version of the Elo system processed games in the order they appeared in the dataset, which was not strictly chronological. A Monday Night Football game was sometimes processed before a Sunday game from the same week, meaning Monday's result influenced Sunday's predictions. Adding a chronological ordering check caught this immediately.

Feature normalization was another subtle problem. Normalizing features across the full dataset means the model has seen the distribution of future data. I implemented expanding-window normalization that only uses data available up to each prediction point, with prior-season bootstrap to handle the cold start at the beginning of each season.

The walk-forward backtest revealed that raw model predictions underperform market prices on average — the market is efficient. The value comes from combining both signals. The blended predictions beat both the pure model and pure market on closing line value, which confirmed the architecture's design hypothesis.

## Results

The system backtests across four NFL seasons with per-season model retraining, producing Brier score decomposition, closing line value metrics, and betting simulations with flat-stake and quarter-Kelly strategies. The production pipeline runs automatically on Fridays via GitHub Actions, timed to the NFL's odds snapshot schedule. The dashboard serves game predictions, performance analytics, and full backtest visualizations with interactive Plotly charts. The codebase includes 55 test files covering the data pipeline, feature builders, model training, and API endpoints.
