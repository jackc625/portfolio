# PRD: Personal Portfolio Website for SWE Job Search

## 1. Product Overview

Build a polished multi-page personal portfolio website for **Jack Cutrara** that helps him secure **junior / entry-level software engineering interviews** by clearly presenting who he is, what he has built, and why he is worth interviewing.

The website should function as a professional online presence that complements a resume by showcasing projects, technical ability, motivation, and overall polish in a way that a static resume cannot.

## 2. Primary Goal

The primary goal of the website is to increase the likelihood of interview invitations for junior / entry-level software engineering roles.

## 3. Secondary Goals

- Help recruiters quickly understand who Jack is
- Showcase technical projects in a compelling and organized way
- Provide easy access to resume and contact information
- Demonstrate professionalism, taste, and attention to detail
- Support both recruiter-friendly browsing and deeper technical review

## 4. Target Audience

### Primary audience

- Recruiters
- Hiring managers
- Engineers involved in the hiring process

### Secondary audience

- Anyone evaluating Jack as a candidate for a software engineering role

## 5. Candidate Positioning

The website should position Jack as:

- a serious junior candidate
- someone who builds real projects
- someone with strong initiative and technical curiosity
- someone who is polished, thoughtful, and employable
- someone capable of backend and full-stack work

The exact personal branding language can be finalized later.

## 6. Product Vision

The portfolio should feel:

- impressive
- clean
- professional
- modern
- personal
- high-end
- polished

It should avoid feeling generic, sloppy, overly flashy, or template-like.

## 7. Site Type

This will be a **multi-page website**.

## 8. Core Pages

The following pages are in scope for v1:

### Home

Primary landing page that quickly communicates who Jack is, what he builds, and why the visitor should keep exploring.

### About

A more detailed page about Jack’s background, education, interests, and path into software engineering.

### Projects

A browsable page showing Jack’s projects, with each project able to link to a detailed breakdown.

### Resume

A page for viewing and downloading Jack’s resume.

### Contact

A page or section that gives visitors clear ways to reach Jack.

## 9. Recommended Additional Page

Not required, but strongly recommended:

### Project Detail / Case Study Pages

Each important project should have its own detailed page. This fits the requirement that project pages should be detailed and allows the site to appeal to both recruiters and engineers.

## 10. Core User Needs

### Recruiter needs

- Understand who Jack is quickly
- See proof of technical ability
- Access resume easily
- Find contact information easily
- Get a positive, polished first impression

### Hiring manager / engineer needs

- Understand what Jack has built
- Assess technical depth and complexity
- Review engineering decisions and project scope
- Evaluate seriousness and initiative

## 11. Key User Stories

- As a recruiter, I want to understand this candidate in under a minute so I can decide whether to continue reviewing.
- As a hiring manager, I want to see meaningful projects that demonstrate software engineering ability.
- As an engineer, I want to explore detailed project pages to understand the technical work.
- As a visitor, I want easy access to Jack’s resume and contact information.
- As Jack, I want the site to present me as a polished and credible junior engineer.

## 12. Functional Requirements

### 12.1 Home Page

The Home page should:

- clearly display Jack Cutrara’s name
- communicate that he is a junior / entry-level software engineering candidate
- introduce him briefly
- highlight featured projects
- provide clear navigation to other pages
- include prominent links to resume and contact information
- make a strong first impression within seconds

### 12.2 About Page

The About page should:

- provide a fuller background/story
- include school information
- present Jack in a professional but human way
- support recruiter understanding beyond project lists
- optionally include interests, motivation, and career direction

### 12.3 Projects Page

The Projects page should:

- display all key projects
- present project summaries in a clean, scan-friendly format
- link to detailed project pages
- support both quick browsing and deeper exploration

### 12.4 Project Detail Pages

Each project detail page should support:

- project overview
- problem or purpose
- tech stack
- features
- implementation details
- technical challenges
- architecture or system design discussion where relevant
- lessons learned
- links to repo and/or demo if available

Detailed content can be added later, but the requirement should exist now.

### 12.5 Resume Page

The Resume page should:

- display resume content or a polished preview
- allow resume download as PDF
- make it easy for recruiters to access without friction

### 12.6 Contact Page

The Contact page should:

- provide clear contact methods
- include professional links such as email, LinkedIn, and GitHub
- make contacting Jack straightforward

## 13. Content Requirements

The site must eventually include:

- name
- role / positioning statement
- education details
- resume
- project summaries
- detailed project breakdowns
- contact information

The following content can be finalized later:

- exact bio
- personal branding statement
- project-specific copy
- about-page narrative
- final headlines and CTA text

## 14. Information Architecture

A likely structure:

- Home
- About
- Projects
  - Project Detail Page
  - Project Detail Page
  - Project Detail Page
- Resume
- Contact

## 15. Design Requirements

The design should prioritize:

- polished visual presentation
- modern layout and typography
- strong spacing and hierarchy
- clean project presentation
- premium feel without being excessive
- clarity for recruiter scanning
- professionalism over gimmicks

The design should balance:

- personal warmth
- technical credibility
- strong aesthetics
- ease of use

## 16. UX Requirements

The site should be:

- easy to scan quickly
- easy to navigate
- mobile responsive
- fast-loading
- visually consistent
- not cluttered
- intuitive for non-technical visitors
- deep enough for technical visitors

## 17. Content Strategy

The website should support two reading modes:

### Fast scan mode

For recruiters who spend less than a minute:

- clear summary
- strong homepage
- obvious featured projects
- accessible resume
- visible contact info

### Deep review mode

For engineers or interested hiring managers:

- detailed project pages
- technical breakdowns
- architecture discussion
- engineering decisions and lessons learned

## 18. Non-Functional Requirements

The site should aim for:

- responsive design across desktop and mobile
- clean and professional UI
- accessibility best practices
- strong performance
- clear SEO/meta structure
- easy maintainability and future updates

## 19. v1 Scope

### In scope for v1

- multi-page website
- Home page
- About page
- Projects page
- detailed project pages
- Resume page
- Contact page
- polished professional design
- placeholders where needed for content to be written later

### Out of scope for now

- blog
- testimonials
- CMS
- analytics-heavy dashboards
- extra personal branding experiments

## 20. Suggested Optional Features for Later

These are not required for the initial PRD but may be considered later:

- featured project section on homepage
- project tags / filtering
- “currently building” section
- GitHub / LinkedIn links in navbar or hero
- case-study template system
- optional blog or writing section

## 21. Success Criteria

The site will be successful if it:

- makes Jack look more credible than a resume alone
- clearly showcases project work
- helps visitors understand him quickly
- supports technical evaluation through detailed project pages
- contributes to improved confidence when applying to jobs
- increases the chances of recruiter engagement and interview requests

## 22. Risks / Open Items

These items can be resolved later and do not block the PRD:

- final personal branding statement
- exact homepage headline
- exact project inventory
- which projects are featured vs secondary
- final visual references
- exact copy for About page
- contact method preferences
- whether any extra pages should be added

## 23. Product Principle

The portfolio should answer this question for a hiring team:

**"Why should we interview Jack instead of treating him like just another entry-level applicant?"**

The site should do that by showing:

- real project work
- seriousness
- polish
- technical depth
- initiative
