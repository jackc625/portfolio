## Overview

Phase 12 is a surgical cleanup pass. Every decision is locked. The only research questions worth asking are the narrow *how* questions — exact schema paths, browser-support percentages, command-line invocations, shared-helper signatures, document skeletons, regression checklists.

**Single most load-bearing finding:** The current `wrangler.jsonc` block is *doubly* wrong — the key is `ratelimits` (no underscore) and the inner field is `name` not `binding` per the [official Cloudflare Workers rate-limit binding docs](https://developers.cloudflare.com/workers/runtime-apis/bindings/rate-limit/). Wrangler 4.36+ recognizes `ratelimits`; we're on 4.80 and current latest is **4.83.0** [verified: `npm view wrangler version` → `4.83.0`]. That means one of two outcomes is correct:

1. **Fix the schema** — rename `rate_limits` → `ratelimits`, rename `binding` → `name` inside the entry. The warning disappears *and* the Cloudflare rate limit binding would actually register (it currently doesn't — the block is inert, which is why app-layer rate limiting in `api/chat.ts` is what actually protects the endpoint).
2. **Delete the block entirely** — it has never worked; app-layer rate limiting is already the enforcement path; removing the block is the cleanest resolution.

Given the explicit D-05 language ("we're already rate-limiting in application code"), **recommendation: delete the block**. Leaving a correctly-named `ratelimits` binding just to silence a warning would give the reader the false impression that Cloudflare-level rate limiting is live when it isn't wired up server-side.

**Primary recommendation:** Upgrade Wrangler 4.80 → 4.83 (devDep), delete the `rate_limits` block from `wrangler.jsonc`, apply the ESLint flat-config `ignores` update, and sequence the rest of the phase so D-26 Chat Regression Gate runs *once* at the end covering all chat-adjacent edits (copy-button helper + ChatWidget inert extension).

## Technical Approach

### Q1 — Wrangler `rate_limits` schema (D-05)

**Finding:** The current block is syntactically wrong on two counts — field name and inner key. Wrangler silently ignores it, emits "Unexpected fields found in top-level field: \"rate_limits\"" once per environment resolution pass (×6 during a full build).

**Official schema (Wrangler 4.36+, verified via Context7 → Cloudflare docs):**

```jsonc
{
  "ratelimits": [
    {
      "name": "MY_RATE_LIMITER",           // not "binding"
      "namespace_id": "1001",
      "simple": {
        "limit": 100,
        "period": 60                         // must be 10 or 60
      }
    }
  ]
}
```

**Evidence trail:**
- Top-level key is `ratelimits` (no underscore) — `"ratelimits": [...]` array. [VERIFIED: Cloudflare Workers docs — Rate Limit binding, fetched 2026-04-15]
- Inner field is `name`, not `binding`. [VERIFIED: same source, both JSON and TOML examples]
- Minimum Wrangler version: 4.36. [VERIFIED: Cloudflare Workers docs]
- Current repo Wrangler: 4.80 (from `package.json` devDep `"wrangler": "^4.80.0"`). Latest npm: 4.83.0. [VERIFIED: `npm view wrangler version`]
- Both 4.80 and 4.83 accept `ratelimits`; neither accepts `rate_limits`. The 4.80→4.83 upgrade is *not required* to fix the schema — it's an independent hygiene upgrade (D-05 informational notice).
- The binding type historically shipped as `[[unsafe.bindings]] type = "ratelimit"` in pre-4.36 wrangler; it graduated out of unsafe bindings into a top-level `ratelimits` key. [VERIFIED: cross-referenced Cloudflare Community thread + docs]

**Recommendation:** **Delete the block.** Rationale:
- App-layer rate limiting lives in `src/pages/api/chat.ts` (CHAT_RATE_LIMITER binding usage per Phase 7 VERIFICATION D-10/D-24). That's the enforcement path either way — the wrangler.jsonc binding has never been wired into runtime code via `env.CHAT_RATE_LIMITER.limit()`.
- Actually — re-check: Phase 7 VERIFICATION line 138 says "chat.ts: `env.CHAT_RATE_LIMITER.limit()`". If that call path exists, the binding *is* needed and the fix is to rename the schema, not delete. **Planner MUST verify via grep before choosing delete vs fix.** See Open Questions.

**Verification test:** Run `pnpm build 2>&1 | grep -c "Unexpected fields"` — must be 0 after the fix.

### Q3 — ESLint ignore syntax (D-07)

**Finding:** The repo uses **flat config** (`eslint.config.mjs`, ESLint 10.1.0) with the file already wiring a top-level `ignores` entry. [VERIFIED: read `eslint.config.mjs:5-7`]

Current shape:
```js
export default [
  {
    ignores: [".astro/", "dist/", ".claude/", ".ship-safe/"],
  },
  ...tseslint.configs.recommended,
  ...eslintPluginAstro.configs.recommended,
  // rules block…
];
```

**Exact one-liner:** Add `"worker-configuration.d.ts"` to the existing `ignores` array — no new config object, no `globalIgnores()` helper needed.

```js
ignores: [".astro/", "dist/", ".claude/", ".ship-safe/", "worker-configuration.d.ts"],
```

**Why this works:** In flat config, a config object that contains *only* `ignores` (no `rules`, no `files`) is a global ignore. Global ignore patterns match file names anywhere in the project root. The file is at repo root (generated by `wrangler types` during `pnpm build`), so the bare name suffices. [VERIFIED: ESLint flat-config ignores docs]

**Alternative (not recommended):** `globalIgnores(["worker-configuration.d.ts"])` via `defineConfig` helper — adds a dependency on `eslint/config` and complicates the already-simple array export. Skip it.

**Verification test:** After the change, `pnpm lint 2>&1 | grep -E "worker-configuration|unused-disable-directive" | wc -l` must return 0.

### Q4 — OG URL verification method (D-18)

**Finding:** The three official validator UIs (Facebook Sharing Debugger, LinkedIn Post Inspector, X Cards Validator) are all manual, single-URL, browser-based, and not scriptable for a 5-URL sweep. X Cards Validator UI was [removed in 2023](https://blog.x.com/); the API still exists but requires OAuth. The reliable scriptable path is **curl with the `facebookexternalhit/1.1` user agent** — Facebook's crawler reads raw `<meta property="og:*">` tags, which is also what LinkedIn (`LinkedInBot/1.0`) and X (`Twitterbot/1.0`) parse. The HTML payload is identical across crawlers for this site (static Astro output), so one curl + one crawler proves behavior for all three.

**Primary scripted method:**

```bash
# 5-page verification sweep
for path in "/" "/about" "/projects" "/projects/seatwatch" "/contact"; do
  echo "=== ${path} ==="
  curl -sS -A "facebookexternalhit/1.1 (+http://www.facebook.com/externalhit_uatext.php)" \
    "https://jackcutrara.com${path}" \
    | grep -E '<meta property="og:(url|image)"' \
    | sed 's/^[[:space:]]*//'
done
```

Expected output per page: exactly one `og:url` and one `og:image`, both absolute URLs beginning with `https://jackcutrara.com/` (for root-relative images) or any other valid `https://` origin. No `https://jackcutrara.comhttps://` double-origin corruption. No empty content.

**Manual fallback (visual proof):**

1. **Facebook Sharing Debugger** — `https://developers.facebook.com/tools/debug/?q=<url>` — canonical source of truth for what Facebook renders; cached, so use "Scrape Again" button after any fix.
2. **LinkedIn Post Inspector** — `https://www.linkedin.com/post-inspector/inspect/<url>` — second-source verification; also forces cache refresh.
3. **X Cards (API)** — `curl -A "Twitterbot/1.0" <url> | grep "twitter:"` — structurally identical to the primary method but with the X crawler UA.

**Record format for `12-VALIDATION.md`:** 5-row table — columns: Path, `og:url` captured, `og:image` captured, Absolute-URL verdict ✓/✗, Notes. Commit the raw curl output alongside as evidence.

**Recommendation:** Primary = curl script above. Fallback = one Facebook Debugger screenshot per page for the record. Do not rely on any of the third-party multi-platform debuggers (microlink, socialcal, socialpreviewhub) — they add an uncontrolled intermediary and their results aren't reproducible from the commit.

### Q5 — Copy-button shared helper (D-02)

**Finding:** After reading both paths verbatim, the divergence surface is precisely:

| Aspect | Live-stream (`chat.ts:285-308`) | Replay (`chat.ts:553-569`) | Reconciliation |
|--------|--------------------------------|----------------------------|----------------|
| Class | `chat-copy-btn` | `chat-copy-btn label-mono` | Use replay (`label-mono`) |
| Inner content | SVG clipboard icon via `innerHTML` | `textContent = "COPY"` | Use replay (COPY text) |
| `type` attr | — | `"button"` | Add `type="button"` |
| Inline `style` | 10 lines incl. width/height/flex | 4 lines (position/background/border/cursor) | Use replay (no fixed size — lets text size dictate) |
| aria-label | `"Copy message"` | `"Copy message"` | Identical |
| Click handler behavior | Only `copyToClipboard(content, copyBtn)` — no post-click visual change | `copyToClipboard(...)` + `textContent = "COPIED"` + accent color + 1s revert | Use replay (full transition) |
| Content source | `content` (closure over function arg) | `capturedContent` (closure over `msg.content`) | Accept parameter |

**Important nuance — the post-stream re-wire at `chat.ts:822-832`:** Live-stream has a second-phase rebind where `copyBtn.replaceWith(copyBtn.cloneNode(true))` strips the in-stream listener, then a new listener is attached using the *final* `botContent` (after all SSE tokens are consumed). That idempotency guard must survive — the helper must support being called with content that may change after creation *or* be called twice (once in-stream with partial content, once post-stream with final content). The planner's decision: either (a) call the helper once post-stream (simpler — but during streaming the button would be absent), or (b) call the helper twice and accept the `cloneNode` replace pattern.

**Proposed helper signature:**

```ts
/**
 * Create the canonical bot-message copy button.
 * Single source of truth for markup/classes/style/behavior used by both
 * the live-stream (createBotMessageEl) and localStorage-replay paths.
 *
 * D-02: replay markup is canonical — COPY/COPIED label, label-mono class,
 * accent-color flip on success, type="button", minimal inline style.
 */
function createCopyButton(getContent: () => string): HTMLButtonElement {
  const btn = document.createElement("button");
  btn.className = "chat-copy-btn label-mono";
  btn.type = "button";
  btn.textContent = "COPY";
  btn.setAttribute("aria-label", "Copy message");
  btn.style.cssText =
    "position: absolute; top: -4px; right: 0; background: none; border: none; cursor: pointer;";
  btn.addEventListener("click", () => {
    copyToClipboard(getContent(), btn);
    btn.textContent = "COPIED";
    btn.style.color = "var(--accent)";
    setTimeout(() => {
      btn.textContent = "COPY";
      btn.style.color = "var(--ink-faint)";
    }, 1000);
  });
  return btn;
}
```

**Key design choice — `getContent` as a function, not a string.** This resolves the mid-stream vs post-stream ambiguity without the `cloneNode` dance. Live-stream passes `() => botContent` (closure over the growing string); replay passes `() => msg.content` (closure over an immutable string). The live path's `chat.ts:822-832` idempotency rewire can be *simplified* — no need to `replaceWith(cloneNode(true))` because the existing listener already reads the latest content on click. (Planner's call whether to simplify in this phase or leave it for later.)

**Placement:** Adjacent to `createBotMessageEl` in `chat.ts` (around line 265-ish, a module-scoped function just above it). No export — this is module-internal.

**Call sites after refactor:**
- `createBotMessageEl` replaces lines 285-308 with `const copyBtn = createCopyButton(() => content);`
- Replay block replaces lines 553-569 with `wrapper.appendChild(createCopyButton(() => msg.content));`

**D-26 gate implication:** This helper IS the chat.ts normalization — D-26 regression battery MUST run after the refactor. Specifically watch: clipboard idempotency (JS boolean in `copyToClipboard` + DOM data-attribute), aria-label unchanged, XSS-sanitized HTML in bubble sibling unaffected.
