<task type="auto">
  <name>Task 1: Fix theme persistence across View Transition navigation</name>
  <files>src/layouts/BaseLayout.astro</files>
  <action>
    In `src/layouts/BaseLayout.astro`, extend the existing `is:inline` script in `<head>` (lines 39-46) to add an `astro:after-swap` event listener. This listener fires BEFORE the new page is painted during View Transition navigation — the correct lifecycle hook for restoring HTML attributes that get replaced during swap.

    Add the following inside the existing `is:inline` script block, after the IIFE that handles initial page load:

    ```javascript
    document.addEventListener('astro:after-swap', function() {
      var stored = localStorage.getItem('theme');
      if (stored === 'light' || (!stored && window.matchMedia('(prefers-color-scheme: light)').matches)) {
        document.documentElement.setAttribute('data-theme', 'light');
      }
    });
    ```

    Important constraints:
    - This MUST remain in the `is:inline` script in `<head>` (not a module script, not in body). The `is:inline` script persists across View Transition navigations — it only runs once on initial load, but the event listener it registers stays active.
    - Do NOT move or modify the existing IIFE — it correctly handles the initial page load case. The `astro:after-swap` listener handles subsequent client-side navigations.
    - Do NOT use `astro:page-load` for this — it fires AFTER paint, causing a flash of wrong theme.
    - Use `var` (not `const`/`let`) to match the existing script's style and avoid strict mode issues in inline scripts.

    Root cause reference: Debug session `.planning/debug/theme-not-persisting-view-transitions.md` — Astro View Transitions replaces `<html>` attributes during swap, losing `data-theme="light"`. The `is:inline` IIFE only runs on initial parse, not during View Transition navigation.
  </action>

    Specific changes:

    1. **Get a reference to the parent section.** After getting the canvas element (~line 28), also get its parent section:

    3. **Change the listener attachment** (~line 62): Replace `canvas.addEventListener('mousemove', onMouseMove)` with `section.addEventListener('mousemove', onMouseMove)`.

    4. **Update the cleanup function** (~line 207): Replace `canvas.removeEventListener('mousemove', onMouseMove)` with `section.removeEventListener('mousemove', onMouseMove)`.

    Do NOT add `pointer-events: none` to the overlay div — that would break clickability of any content inside the slot (like CTAs or links in the hero area). Moving the listener to the section is the correct approach.

    Root cause reference: Debug session `.planning/debug/canvas-hero-mouse-influence.md` — overlay div at z-10 intercepts all pointer events before they reach the canvas. Canvas mousemove listener never fires.
