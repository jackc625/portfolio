For each site, extract: (1) how "about" content is presented (not as a plain bio wall), (2) compositional techniques for text-heavy pages, (3) how skills/capabilities are shown without generic card grids.

**DESIGN PRINCIPLES extracted from inspiration sites (apply these):**

1. **Narrative as visual composition** (from artemshcherban.com, bettinasosa.com): The about page is not "heading + paragraphs + skills grid." The text itself creates visual rhythm through varied typography scales, strategic line breaks, and spacing. A lead paragraph in display type, followed by body text at a different horizontal position, creates editorial flow.

2. **Asymmetric two-register layout** (from aither.co, shiyunlu.com): Instead of single-column prose, use a layout where different content zones occupy different spatial positions. Example: biographical narrative in a narrow column (40ch-50ch wide) positioned at 2/3 from the left edge, while metadata/facts appear as small-type annotations at the left margin. Or: alternating between wide and narrow content widths within the same page.

3. **Skills as typography, not cards** (from wam.global, artemshcherban.com): Instead of 4 identical bg-bg-secondary cards in a 2x2 grid, present skills as a typographic element. Options:
   - Large display-type skill categories as section dividers with skills listed inline below each
   - A single-column flowing list where category names are in mono and skills flow as comma-separated body text
   - Skills integrated into the narrative itself (mentioned in context rather than in a separate section)
   - An editorial list with alternating alignment — category on left, skills on right, separated by a thin rule

4. **Page-level horizontal variation** (from aither.co): The about page should NOT use the same horizontal padding/alignment for every section. The hero text might span wide, the narrative might be in a narrow column offset right, and the skills might use a different grid altogether.

5. **Strategic whitespace as design element**: Large vertical gaps between sections create breathing room and editorial pacing. Use at least 120px-200px gaps between major content blocks at desktop.

**LAYOUT CONCEPT TO IMPLEMENT:**

**Hero/opening area:**
- Opening headline at display scale, positioned asymmetrically (not centered, not at the same left edge as body text)
- The headline should be a compelling statement (e.g., "Building things is how I learn.") not just the word "About"
- Below the headline, a small metadata strip (location, status) in mono text — positioned at a different horizontal alignment than the headline
- The heading uses `font-size: clamp(2.5rem, 1.5rem + 8vw, 8rem)` with tight line-height (0.9) and negative tracking

**Narrative section:**
- Biographical prose in a narrow column (max-width: 38rem or 45ch) positioned with an asymmetric left offset
- Lead paragraph in display type (larger, serif) to create hierarchy
- Remaining paragraphs in body type (smaller, sans)
- Use a colored accent divider (thin horizontal line in accent color) between the lead and body paragraphs
- A pull-quote or closing statement highlighted with a left border accent
- On desktop, use a sidebar column with small metadata (focus area, education, interests) at a fixed position while the narrative scrolls — this creates spatial depth

**Skills section:**
- DO NOT use 4 identical SkillGroup cards in a 2x2 grid — this is what was rejected
- Instead, present skills as an editorial typographic element. Choose ONE of these approaches:
  a) **Inline-annotated list**: Each skill category as a display-scale word (e.g., "Languages" at 2rem, serif) followed by skills as body text on the same line or directly below, separated by thin horizontal rules. Left-aligned, no cards, no backgrounds.
  b) **Margin-annotated layout**: Skill categories as small mono labels in a left margin column, with actual skills as body text in the main content column — like editorial footnotes.
  c) **Flowing prose approach**: A single paragraph or two that mentions skills in context: "I work primarily with TypeScript and Python, building with React and Next.js on the frontend..."
- The SkillGroup component may still be imported and used if approach (a) or (b) works, but the surrounding layout must NOT be a uniform grid.

**CTA/closing area:**
- Minimal "Get in Touch" and "Resume" links as text (font-display at heading scale)
- Same treatment as the home page quick links — subtle, not boxed

**TECHNICAL REQUIREMENTS:**
- Keep `import BaseLayout` and `import SkillGroup` (SkillGroup may or may not be used depending on design)
- All colors via design tokens
- All fonts via font-display, font-body, font-mono
- Responsive: asymmetric layout on desktop, stacked on mobile
- Maintain aria-labels: "Background" on narrative section, "Skills" on skills section
- Keep the BaseLayout wrapper with `title="About"` and description
- Use `text-text-primary` (not text-text-secondary) for body text on bg-bg-secondary backgrounds per contrast rules

**WHAT TO REMOVE from current page:**
- Film grain overlay (.grain div and its CSS)
- The generic "about-hero" 70vh/80vh section that mirrors the home page hero
- The symmetric 2-column sidebar layout that looks like every other portfolio
- Animation prep classes that hide content by default (opacity: 0) — these make the page invisible without JS
- Complex GSAP animation scripts (simplify to basic fade-in only, or remove and let content be visible by default)

**WHAT TO KEEP:**
- First-person narrative content (all 4 paragraphs — lead, education, interests, closing)
- Skill data (Languages, Frameworks, Tools, Concepts with their items)
- Semantic HTML (sections with aria-labels, h1, h2)
- Professional but human tone per D-05
- The location metadata ("New York, NY") and status ("Open to Opportunities")
- Links to /contact and /resume

**IMPORTANT — Content visibility without JS:**
The current page sets all content to `opacity: 0` via CSS classes (.about-line, .about-fade, .sidebar-item, .narrative-block, .skill-reveal) and relies on GSAP to reveal them. This means the page is INVISIBLE if JS fails or is slow. The redesign MUST ensure all content is visible by default. If animations are desired, use a progressive enhancement approach: content is visible, and JS adds entrance animations on top.
