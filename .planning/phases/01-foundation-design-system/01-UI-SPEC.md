# Phase 1 — UI Design Contract

> Visual and interaction contract for the foundation design system. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> This phase establishes the design token foundation — no pages or interactive components are built here.
> All subsequent phases reference this contract as their visual source of truth.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Astro + Tailwind CSS v4 — not a React SPA, shadcn not applicable) |
| Preset | not applicable |
| Component library | none (vanilla Astro components with Tailwind utilities) |
| Icon library | none for Phase 1 (deferred to Phase 2 when navigation is built) |
| Fonts | Instrument Serif (display) + Instrument Sans (body) + JetBrains Mono (mono) |

### Font Rationale

**Display — Instrument Serif:** Condensed, high-contrast serif with confident curves. Designed by Rodrigo Fuenzalida for the Instrument brand. Available on Google Fonts in Regular and Italic. Not overused like Playfair Display. The condensed letterforms create dramatic, editorial-grade headlines on dark backgrounds.

**Body — Instrument Sans:** Variable sans-serif from the same Instrument family. Balances precision with subtle playfulness. Clean legibility at body sizes. The family pairing with Instrument Serif creates visual cohesion without monotony.

**Mono — JetBrains Mono:** Purpose-built for code readability with programming ligatures. Signals developer credibility to technical reviewers who inspect the portfolio. Available on Google Fonts. Increased height for lowercase letters improves readability at small sizes.

**Why not the reference site fonts:** The frontend-design skill requires avoiding convergence on common choices. Reference sites use Space Grotesk, DM Sans, Amiri, Fragment Mono, Inter — all excellent but already associated with those portfolios. Instrument Serif/Sans is a distinctive, underused pairing that shares the editorial DNA of the reference sites without copying them.

## Spacing Scale

Declared values (all multiples of 4):

| Token | CSS Custom Property | Value | Usage |
|-------|---------------------|-------|-------|
| xs | `--token-space-xs` | 4px (0.25rem) | Icon gaps, inline padding, tight element margins |
| sm | `--token-space-sm` | 8px (0.5rem) | Compact element spacing, tag padding |
| md | `--token-space-md` | 16px (1rem) | Default element spacing, component padding |
| lg | `--token-space-lg` | 24px (1.5rem) | Card padding, form group gaps |
| xl | `--token-space-xl` | 32px (2rem) | Layout gaps between components |
| 2xl | `--token-space-2xl` | 48px (3rem) | Major section breaks within a page |
| 3xl | `--token-space-3xl` | 64px (4rem) | Page-level vertical padding |
| 4xl | `--token-space-4xl` | 96px (6rem) | Hero section vertical padding |
| 5xl | `--token-space-5xl` | 128px (8rem) | Maximum vertical breathing room |
| section | `--token-space-section` | clamp(4rem, 3rem + 5vw, 8rem) | Fluid section padding (responsive) |

Exceptions:
- Touch targets: minimum 44px (accessible tap target per WCAG 2.5.5) — applies to interactive elements in Phase 2+
- The "spacious whitespace density" (D-02) means defaulting to `lg` and `xl` for component padding, not `md`. Components should breathe.

## Typography

### Font Families

| Role | Family | CSS Variable | Provider | Weights | Styles |
|------|--------|-------------|----------|---------|--------|
| Display | Instrument Serif | `--font-heading` | Google Fonts (via Astro Fonts API) | 400 | normal, italic |
| Body | Instrument Sans | `--font-sans` | Google Fonts (via Astro Fonts API) | 400, 500, 600, 700 | normal, italic |
| Mono | JetBrains Mono | `--font-code` | Google Fonts (via Astro Fonts API) | 400, 500 | normal |

### Type Scale

Dramatic type scale with fluid sizing via `clamp()`. Display headings are oversized and editorial — this is the "one thing someone will remember" per the frontend-design skill. The scale uses a ~1.333 (perfect fourth) ratio at mobile and ~1.5 (perfect fifth) ratio at desktop for elegant scaling.

| Role | Token | Min (mobile) | Max (desktop) | CSS Clamp | Weight | Line Height | Letter Spacing |
|------|-------|-------------|--------------|-----------|--------|-------------|----------------|
| Caption | `--token-text-xs` | 12px | 13px | clamp(0.75rem, 0.7rem + 0.25vw, 0.8125rem) | 400 body / 500 mono | 1.5 | 0 / 0.02em mono |
| Small | `--token-text-sm` | 13px | 14px | clamp(0.8125rem, 0.75rem + 0.3vw, 0.875rem) | 400 | 1.5 | 0 |
| Body | `--token-text-base` | 15px | 16px | clamp(0.9375rem, 0.875rem + 0.3vw, 1rem) | 400 | 1.6 | 0 |
| Large body | `--token-text-lg` | 17px | 20px | clamp(1.0625rem, 0.95rem + 0.55vw, 1.25rem) | 400 | 1.5 | 0 |
| Heading 4 | `--token-text-xl` | 20px | 24px | clamp(1.25rem, 1.1rem + 0.75vw, 1.5rem) | 600 sans | 1.3 | -0.01em |
| Heading 3 | `--token-text-2xl` | 24px | 32px | clamp(1.5rem, 1.25rem + 1.25vw, 2rem) | 600 sans | 1.25 | -0.015em |
| Heading 2 | `--token-text-3xl` | 30px | 44px | clamp(1.875rem, 1.5rem + 1.85vw, 2.75rem) | 400 serif | 1.15 | -0.02em |
| Heading 1 | `--token-text-4xl` | 36px | 56px | clamp(2.25rem, 1.75rem + 2.5vw, 3.5rem) | 400 serif | 1.1 | -0.025em |
| Display | `--token-text-5xl` | 48px | 80px | clamp(3rem, 2.25rem + 3.75vw, 5rem) | 400 serif | 1.05 | -0.03em |

### Typography Rules

1. **Display and H1-H2:** Use Instrument Serif at weight 400. The serif's natural contrast and condensed form provide visual weight without needing bold. Negative letter-spacing tightens the editorial feel.
2. **H3-H4:** Use Instrument Sans at weight 600 (semibold). These transitional headings bridge serif display and sans body text.
3. **Body text:** Instrument Sans at weight 400, 16px max at 1.6 line-height. The generous line-height supports the spacious density directive (D-02).
4. **Monospace scope:** JetBrains Mono is used for code blocks, inline code, tech stack tags, metadata labels (dates, categories), and status badges. This dual-purpose usage (code + design accent) reinforces the developer identity throughout the site.
5. **Maximum two weights per font in any single component.** Do not combine 400 + 500 + 600 + 700 in the same visual context.

### Accent Reserved For

The accent color (`--token-accent`, oklch(0.72 0.12 178)) is reserved EXCLUSIVELY for:

1. **Primary CTA buttons** — "View Projects" on home, "View Case Study" on cards
2. **Active navigation indicator** — current page underline/highlight in nav
3. **Links in body text** — inline anchor tags within prose content
4. **Focus ring** — visible keyboard focus indicator (accessibility)
5. **Tech stack tags** — subtle accent border or background tint on technology labels

The accent is NOT used for: headings, body text, borders (use `--token-border`), backgrounds (use `--token-bg-secondary`), icons (use text colors), or decorative elements.

### Contrast Verification Targets

All text/background combinations must meet WCAG AA (4.5:1 for body, 3:1 for large text):

| Foreground | Background | Target Ratio | Expected |
|------------|------------|-------------|----------|
| `--token-text-primary` (0.93 L) | `--token-bg-primary` (0.13 L) | 4.5:1+ | ~13:1 PASS |
| `--token-text-secondary` (0.62 L) | `--token-bg-primary` (0.13 L) | 4.5:1+ | ~5.5:1 PASS |
| `--token-text-primary` (0.93 L) | `--token-bg-secondary` (0.18 L) | 4.5:1+ | ~10:1 PASS |

## Copywriting Contract

Phase 1 is infrastructure — no user-facing pages are built. The copywriting contract defines the copy primitives that downstream phases will use.

| Element | Copy |
|---------|------|
| Primary CTA | "View Projects" (Phase 3 home hero) |
| Empty state heading | "No projects yet" |
| Empty state body | "Projects will appear here once content is added. Check back soon." |
| Error state | "Something went wrong loading this content. Try refreshing the page." |
| Destructive confirmation | Not applicable in Phase 1 (no destructive actions exist) |
| Schema validation placeholder tagline | "A brief project summary goes here" |
| Schema validation placeholder description | "This project demonstrates technical problem-solving and modern development practices. Full case study coming soon." |

### Tone Guidelines

Per ABUT-02 and the dark editorial minimal direction:
- **Professional but human** — not corporate jargon, not overly casual
- **Confident and direct** — active voice, specific verbs, no hedging
- **Concise** — respect the recruiter's 30-second scan; every word earns its place
- **Technical when appropriate** — do not dumb down for engineers; do not overload for recruiters

## Phase 1 Specific: Token Architecture

### CSS Custom Property Layers

The design token system uses two layers as specified in RESEARCH.md Pattern 2:

**Layer 1: Raw values** (`:root` for dark default, `[data-theme="light"]` for Phase 5)
```
:root {
  --token-bg-primary: oklch(0.13 0.005 260);
  --token-text-primary: oklch(0.93 0.005 260);
  /* ...all tokens from tables above... */
}
```

**Layer 2: Tailwind theme bridge** (`@theme` directive references Layer 1 via `var()`)
```
@theme {
  --color-bg-primary: var(--token-bg-primary);
  --color-text-primary: var(--token-text-primary);
  /* ...semantic mapping... */
}
```

This two-layer architecture means:
- Phase 5 only needs to add `[data-theme="light"]` overrides to Layer 1
- Layer 2 (Tailwind utilities) automatically picks up theme changes
- Zero refactoring needed when light mode is added

### Tailwind Utility Class Mapping

| Design Intent | Tailwind Class | Resolves To |
|---------------|---------------|-------------|
| Page background | `bg-bg-primary` | `--token-bg-primary` |
| Card background | `bg-bg-secondary` | `--token-bg-secondary` |
| Primary text | `text-text-primary` | `--token-text-primary` |
| Secondary text | `text-text-secondary` | `--token-text-secondary` |
| Accent text/border | `text-accent` / `border-accent` | `--token-accent` |
| Display font | `font-display` | Instrument Serif |
| Body font | `font-body` | Instrument Sans |
| Mono font | `font-mono` | JetBrains Mono |
