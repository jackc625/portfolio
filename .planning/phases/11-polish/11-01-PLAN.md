**D-05: Fix lightning-css warnings (4x "Unexpected token Delim('*')").**

The `@source` directive in `src/styles/global.css` line 37 is currently:
```css
@source "../**/*.{astro,html,js,jsx,ts,tsx,md,mdx}";
```

This resolves from `src/styles/` to `src/**/*` which is correct for source files. However, Tailwind v4 ALSO auto-detects template files in the entire repo by default. The `@source` directive ADDS to auto-detection, it does not replace it. The `.planning/` markdown files contain literal class strings like `px-[var(--token-space-*)]` and `text-[length:var(--token-text-*)]` from v1.0 plan artifacts, and Tailwind's auto-detection picks those up.

The fix: Add a `@source not` directive to exclude `.planning/` and `design-system/` directories from Tailwind's auto-detection. Add these lines immediately AFTER the existing `@source` line (line 37):

```css
@source not "../../.planning/**";
@source not "../../design-system/**";
```

Update the comment above the @source block (lines 33-36) to explain both directives:
```css
/* Scope Tailwind class detection to source files only.
 * Tailwind v4 auto-detects template files in the repo. The .planning/ directory
 * contains v1.0 plan artifacts with literal class examples like px-[var(--token-*)]
 * that generate lightning-css parse warnings. The @source not directives exclude
 * .planning/ and design-system/ from Tailwind's detection surface. */
```

**D-07b: Fix Container Props ts(6196) hint.**

In `src/components/primitives/Container.astro`, the `Props` interface at line 14 triggers `ts(6196)` because Astro's type system sometimes doesn't recognize implicit interface consumption. The simplest fix that satisfies the TypeScript checker: add `export` before the interface (Astro recognizes exported Props interfaces as the canonical props type):

Actually, in Astro components the `Props` interface name is special and implicitly consumed. The ts(6196) is a false positive from the language server. The cleanest fix is to suppress it without changing the API. Change the interface block from:
```typescript
interface Props {
  as?: keyof HTMLElementTagNameMap;
  class?: string;
}
```
to:
```typescript
type Props = {
  as?: keyof HTMLElementTagNameMap;
  class?: string;
};
```

Using `type` instead of `interface` typically resolves the ts(6196) hint in Astro components because the type alias is consumed by the destructuring assignment on the next line.

If the hint persists after changing to `type`, instead keep the original `interface Props` but suppress with a `// @ts-ignore` comment above line 14. The priority is hint-free `astro check` output, not purity of approach.
