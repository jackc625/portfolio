# Phase 2 -- UI Design Contract

> Visual and interaction contract for the site shell: header, navigation, mobile menu, footer, SEO meta, and accessibility infrastructure. Generated by gsd-ui-researcher.
>
> This phase builds the persistent chrome that wraps every page. All design tokens are inherited from Phase 1 UI-SPEC. This contract specifies only the Phase 2-specific component behaviors, layout rules, and copy.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Astro 6 + Tailwind CSS v4 -- not React, shadcn not applicable) |
| Preset | not applicable |
| Component library | none (vanilla Astro components with Tailwind utilities) |
| Icon library | Inline SVG -- hand-coded hamburger icon, GitHub/LinkedIn/email SVGs embedded directly. No icon package. |
| Fonts | Instrument Serif (display) + Instrument Sans (body) + JetBrains Mono (mono) -- inherited from Phase 1 |

### Icon Rationale

Phase 2 uses exactly 4 icons: hamburger menu (3 lines), close X, GitHub mark, LinkedIn mark, and an email envelope. This count does not justify an icon library dependency. Inline SVG keeps the bundle at zero JS for icons, allows precise color control via `currentColor`, and avoids external HTTP requests. SVG paths for GitHub and LinkedIn marks are publicly available brand assets. The hamburger and close icons are trivial 3-line/2-line SVGs.

## Spacing Scale

Inherited from Phase 1 UI-SPEC without modification:

| Token | Value | Usage in Phase 2 |
|-------|-------|-------------------|
| xs | 4px (0.25rem) | Gap between hamburger icon lines |
| sm | 8px (0.5rem) | Nav link inline padding, icon button internal padding |
| md | 16px (1rem) | Header horizontal padding (mobile), footer content gaps |
| lg | 24px (1.5rem) | Header horizontal padding (desktop), gap between nav links |
| xl | 32px (2rem) | Footer vertical padding |
| 2xl | 48px (3rem) | Mobile menu link vertical spacing |
| 3xl | 64px (4rem) | Not used in Phase 2 |
| section | clamp(4rem, 3rem + 5vw, 8rem) | Not used in Phase 2 (page section spacing, not shell) |

### Phase 2 Exceptions

- **Header height:** 64px (4rem) fixed height. This is a layout constant, not a spacing token. It sets the fixed positioning offset for `<main>` top padding.
- **Touch targets:** All interactive elements (nav links, hamburger button, footer icons) have a minimum 44x44px tap area per WCAG 2.5.5. The hamburger button visible area is 44x44px. Nav links achieve this via padding.
- **Mobile menu link size:** Links in the full-screen overlay use `--token-text-heading` (20-32px fluid) with 48px vertical padding between them, making each link's tap area well above 44px.

## Typography

Inherited from Phase 1 UI-SPEC. Phase 2 usage mapping:

| Role | Token | Font | Weight | Line Height | Phase 2 Usage |
|------|-------|------|--------|-------------|---------------|
| Label / Caption | `--token-text-sm` (13-14px fluid) | Sans | 400 | 1.5 | Footer copyright text, skip-to-content link |
| Body | `--token-text-base` (15-16px fluid) | Sans | 400 | 1.6 | Desktop nav links, footer social link labels (if visible) |
| Heading | `--token-text-heading` (20-32px fluid) | Sans | 600 | 1.25 | Mobile menu overlay links |
| Display | `--token-text-display` (36-80px fluid) | Serif | 400 | 1.05 | Not used in Phase 2 shell (reserved for page content H1s) |

### Phase 2 Typography Rules

1. **Namemark "Jack Cutrara":** Uses `font-display` (Instrument Serif) at `--token-text-base` size (15-16px), weight 400. The serif typeface provides visual distinction from the sans-serif nav links without needing a larger size. It is NOT a heading element -- it is an `<a>` tag.
2. **Desktop nav links:** `font-body` (Instrument Sans), `--token-text-sm` size (13-14px), weight 400, uppercase with `letter-spacing: 0.05em`. The uppercase treatment at small size creates a clean, editorial navigation bar that does not compete with page headings.
3. **Mobile menu links:** `font-body` (Instrument Sans), `--token-text-heading` size (20-32px fluid), weight 600. Large, centered, generous spacing -- dramatic full-screen feel per D-05.
4. **Footer text:** `font-body` at `--token-text-sm` (13-14px), weight 400. Understated.

## Color

Inherited from Phase 1 UI-SPEC. Phase 2 specific usage:

| Role | Token | Phase 2 Application |
|------|-------|--------------------|
| Dominant (60%) | `--token-bg-primary` | Page background behind header, behind main, behind footer |
| Secondary (30%) | `--token-bg-secondary` | Header scrolled-state background (with 85% opacity), mobile menu overlay background, footer background |
| Accent (10%) | `--token-accent` | Active nav link indicator, focus rings, footer link hover |
| Accent hover | `--token-accent-hover` | Active nav link hover state |

### Phase 2 Color Rules

1. **Header at top of page (not scrolled):** Fully transparent background (`background: transparent`). Text uses `--token-text-primary`. Blends into dark page content per D-01.
2. **Header after scroll (scrolled state):** `--token-bg-secondary` at 85% opacity with `backdrop-filter: blur(12px)`. This provides contrast/readability per D-01 while maintaining the editorial glass effect seen in reference sites.
3. **Mobile menu overlay:** `--token-bg-primary` at 98% opacity. Nearly opaque to prevent background content from distracting. A slight transparency maintains depth.
4. **Footer:** `--token-bg-secondary` as solid background. Creates a visual boundary between page content and footer without a visible border line.
5. **Active nav link:** `--token-accent` text color. Inactive links use `--token-text-secondary` to de-emphasize, with hover state transitioning to `--token-text-primary`.
6. **Focus rings:** `ring-2 ring-accent outline-none` on all interactive elements when focused via keyboard (`:focus-visible`).

### Accent Reserved For (Phase 2 Specific)

Within this phase, accent color appears ONLY on:
1. Active page nav link text (desktop and mobile)
2. Keyboard focus ring on all interactive elements
3. Skip-to-content link text (when focused/visible)
4. Footer social icon hover state

Accent is NOT used on: the hamburger icon, the namemark, inactive nav links, the footer background, borders/dividers, or the close button in the mobile menu.

## Interaction Contracts

### Scroll-Reveal Header

| State | Trigger | Visual Result |
|-------|---------|---------------|
| Top of page | `scrollY <= 50` | Transparent background, fully visible |
| Scrolled | `scrollY > 50` | Semi-transparent bg-secondary background with blur |
| Scrolling down | `scrollY > 100` and `currentY > lastY` | `translateY(-100%)` -- header slides up and hides |
| Scrolling up | `currentY < lastY` | `translateY(0)` -- header slides back down |
| Mobile menu open | Menu trigger clicked | Header stays visible (z-index above menu ensures this) |

Implementation: Vanilla JS `<script>` with `requestAnimationFrame` throttling. Passive scroll listener. Re-initialized on `astro:page-load`.

### Mobile Menu Open/Close

| State | Trigger | Visual Result |
|-------|---------|---------------|
| Closed | Default | Menu hidden, body scrollable, hamburger icon shown |
| Opening | Hamburger click | Menu fades in (200ms), links stagger up (300ms + 50ms/link), body locked, hamburger animates to X |
| Open | Stable state | Focus trapped in menu, Escape key listens |
| Closing (link) | Nav link click | Menu immediately hides (no exit animation -- instant navigation feels snappier) |
| Closing (escape) | Escape key | Menu fades out (150ms), focus returns to hamburger, body unlocked |
| Closing (button) | X button click | Same as escape close |
| After navigation | `astro:page-load` fires | Menu state reset to closed (prevents stale state after View Transitions) |

### Active Page Indication

| Location | Active Style | Inactive Style |
|----------|-------------|----------------|
| Desktop nav | `text-accent` | `text-text-secondary` hover `text-text-primary` |
| Mobile menu | `text-accent` | `text-text-primary` hover `text-accent` |
| Detection | `Astro.url.pathname` server-side comparison | N/A |
| Home special case | Exact match `pathname === "/"` | Other pages use `pathname.startsWith(href)` |

### Keyboard Navigation Flow

Tab order through the page:

1. Skip-to-content link (visually hidden, appears on focus)
2. Namemark home link
3. Desktop nav links (Home, About, Projects, Resume, Contact) OR hamburger button (mobile)
4. `<main id="main-content">` content
5. Footer social links (GitHub, LinkedIn, Email)
6. Footer name link (if present)

When mobile menu is open:
1. Close button (receives initial focus)
2. Menu nav links (Home, About, Projects, Resume, Contact)
3. Wraps back to close button (focus trap)

## Accessibility Contract

| Requirement | Implementation |
|-------------|----------------|
| SEOA-04: Semantic HTML | `<header>`, `<nav>`, `<main>`, `<footer>` landmark elements. Single `<h1>` per page. |
| SEOA-05: Keyboard navigation | Tab order: skip link > namemark > nav/hamburger > main > footer. Visible `focus-visible` rings. |
| SEOA-06: Skip-to-content | `<a href="#main-content">` as first focusable element. `sr-only` default, visible on focus. |
| SEOA-07: Alt text / ARIA | SVG icons use `aria-hidden="true"` with adjacent visible text, or `aria-label` for icon-only buttons. |
| IDNV-03: Mobile menu a11y | Focus trap, Escape to close, `aria-expanded`, `aria-controls`, `role="dialog"`, `aria-modal="true"`. |
| Focus indicators | `focus-visible:ring-2 focus-visible:ring-accent focus-visible:outline-none` on all interactive elements. |
| Color contrast | All text/background pairs meet WCAG AA (verified in Phase 1 UI-SPEC contrast table). |
